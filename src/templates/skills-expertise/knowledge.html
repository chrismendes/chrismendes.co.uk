Responsive Development

With a dozen responsive sites under my belt I am very well versed in the issues surrounding responsive web design.

Using off-the-shelf grid systems such as Twitter Bootstrap and Foundation, or my new favourite Susy, I can implement x-column grids as required.

I also have experience with solutions such as lazy loading, off canvas menus, and responsive images.

And I'm an advocate for mobile first design/development when project circumstances allow it.

Browser Support

Having often been required to accommodate browsers as old as IE7, I have acquired a very good understanding of what's generally supported and what's not.

I prefer to adopt a progressive enhancement approach to CSS so as to guarentee basic presentational correctness in all browsers from the lowest upwards.

Enhanced presentation can then be delivered to modern browsers with the proper CSS overrides alone, or in conjunction with the JS library Modernizr for feature detection purposes that I am also proficient with.

SMACSS

https://smacss.com

Scalable and Modular Architecture for CSS is an excellent design pattern for CSS authored by Jonathan Snook (ex-Yahoo Front-end Engineer).

Having read his book and followed his online course, and having applied SMACSS commercially since, I adopt this approach to CSS in every greenfield project that I am assigned to.

CSS written within the SMACSS pattern is clean, efficient, easily reusable, and easy to maintain between present and future developers.

Object-Oriented JavaScript

First studied object-oriented programming as an undergraduate at University working with Java, learning the concepts of inheritance, encapsulation, polymorphism, method overloading/overriding, etc.

This vital learning process early on has enabled me ever since to understand newly encountered OOP implementations very quickly.

While JavaScript of course features a different kind of inheritance, namely prototypal, nevertheless I've always preferred when appropriate to code in this natural declarative way.

Design Patterns

A huge interest of mine, my current grasp of this subject matter means that my JavaScript code is always based on an explicitly deliberated organisational structure and often formal design pattern, such as the revealing module pattern.

Understanding the MVC pattern and its MV* variations also means that abiding by the separation of concerns principle when working with JavaScript frameworks is a mere formality.

In the future I intend this to be specialist subject of mine, acquiring a comprehensive insight on the full range of known application architecture and class instantiation design patterns.

Script Loaders

Although not always necessary, and sometimes an "overkill" solution in some circumstances, I realise the benefits a script loader can bring to a project related to either code organisation or performance for example.

Familiar and experienced with Require.js, I've recently been experimenting with non-AMD alternatives such as LabJS and Head.js.

Git Repositories

Have used Git as my predominant source control system for the past few years.

I am familiar with commonly adopted best practices such as the Gitflow team workflow pattern - feature branches, release branches, hotfixes, etc. - and the need for pull requests and good communication in teams.

Preferring to use the command line over tools such as SourceTree, I am familiar with a wide range of Git commands.

Modern Workflow

A keen interest of mine, I always leave the door open to discover new tools and techniques to further assist and automate developer workflow.

Often deployed via the task runner Grunt, solutions range from auto-reloading the browser (LiveReload, BrowserSync), automatically generating spritesheets as required, and building the entire app into an optimised production-ready state.

Also a big fan of the Emmet toolkit, as well as other various plugins for my favourite code editor Sublime Text!

Chrome DevTools

An essential part of any developer's toolkit, I've made it my goal to know about most of what DevTools provides rather than just the minimum.

Ranging from the DOM/styles inspector and the console, to the sources tab for debugging JavaScript, and the network, timeline, and profiles tabs for analysing performance.

Performance - Basic

Like any responsible developer I have a good understanding of the basic measures that should be taken to minimise overhead that require minimal effort only, specifically for example:

  • Minifying CSS and JavaScript
  • Contatenating files to reduce HTTP requests
  • Loading scripts asynchronously to prevent blocking
  • Using spritesheets to reduce requests for images
  • Compressing assets with Gzip on the server
  • ...etc.

I use my task runner of choice Grunt to automate tasks like the above and others.

Performance - Advanced

Here I am referring to two aspects of web performance: cutting-edge modern solutions to bring down page load speeds beyond what basic measures can achieve, and comprehensive background information as to how browsers and web servers work in this area.

This is an ongoing learning process for me and I intend it to be a specialist subject of expertise in the near future.

Currently I am experimenting with how to implement critical path solutions and caching assets with LocalStorage, while also learning about the browser's inherent TCP slow start procedure when requesting/receiving data packets.

Page Speed Analysis

I am familiar with a range of tools that help developers assess the performance of web applications.

Examples include:

  • Google PageSpeed Insights
  • Webpagetest.org
  • Chrome DevTools
  • YSlow

Web performance requirements often vary between projects, in terms of effort priority, but at a minimum I'll often refer to YSlow to check that basic developer measures have been implemented and Google PageSpeed Insights to advise on mobile performance.

Unit Testing

Sound awareness and knowledge of unit testing theory, how to write testable JavaScript, what to test, how to test, and how the running of unit tests can be integrated into build and deployment processes.

With respect to JavaScript I currently resort to Jasmine as my test framework of choice. I also have experience with PHPUnit for testing PHP applications.

In the very near future I'll be adding to my unit testing skillset with the likes of Mocha, Sinon.js, and Karma.

UI Testing

In order to ensure that HTML and CSS renders as expected across a range of browsers, I'm familiar with a number of tools that help with this objective.

Examples include:

  • VirtualBox - to run virtual machines with instances of other operating systems and browser versions.
  • BrowserStack - an online service that provides virtual machines to nearly every OS and browser under the sun.
  • Chrome DevTools - to view the application within a particular viewport for a given phone or tablet device.
  • BrowserSync - a tool that keeps multiple browsers/devices exactly in sync throughout a user's navigation of the app.
